This document will help you get your game up and running quickly with the Big Fish iOS SDK.

Before you begin developing with this version of the Big Fish iOS SDK, review the Release Notes.

  • For an introduction to the SDK, see the SDK Overview.
  • For troubleshooting tips for common development issues, see Frequently Asked Questions.
  • For the most current version of the Big Fish iOS SDK documentation, visit the Big Fish Developers Website.
Note
Discuss with your Big Fish Producer which SDK features you should implement in your game.

Requirements

Development requirements depend on what type of game you are developing, Freemium or Free-to-Play.

  • If you are unsure of the type of game you are developing, review the following documentation:
    • Freemium Game Overview: Introduction to the Freemium game and business model, and required Freemium features
    • Free-to-Play Games Overview: Introduction to the Free-to-Play game and business model, and required Free-to-Play features
  • If you are new to Big Fish SDK development, review the Required Tools and Standards (development tools, third-party tools and libraries, coding standards, device and software support).
  • If you are developing a Freemium game, review the requirements and best practices described in the Freemium Game Development Documentation.
  • If you are developing a Free-to-Play game, review the requirements and best practices described in the Free-to-Play Game Development Documentation.

Integrating the Big Fish iOS SDK into Your iOS Game

Step 1: The Xcode project

If this is a new game, create a new project. For more information, see Xcode requirements.

If you are updating a project for an existing game, you must remove earlier versions of the Big Fish iOS SDK from your project:

  1. Delete BFGLib libraries and resources.
  2. Clean your project in Xcode (in the menu bar, select Product > Clean).
  3. Remove builds with BFGLib from all devices and simulators.
  4. Delete any Flurry library paths or framework paths that you may have previously set.

Step 2: Add required frameworks and libraries

The SDK depends upon the following frameworks and libraries in the iOS Developer Library. Add them to the Frameworks directory in your project before you begin development.

  • CoreGraphics.framework: C-based API based on the Quartz advanced drawing engine.
  • CoreText.framework: API for laying out text and handling fonts.
  • CoreTelephony.framework: Features for obtaining information about a user’s home cellular service provider.
  • Foundation.framework: Defines the base layering of Objective-C classes.
  • GameKit.framework: Features for creating social games.
  • iAd.framework: Provides the classes to deliver ads for iOS.
  • CFNetwork.framework: Provides a library of abstractions for network protocols.
  • MobileCoreServices.framework: Defines the low-level types used in uniform type identifiers (UTIs).
  • libicucore.dylib: The core library for the International Components for Unicode open source project, which enables internationalization and globalization features. The regexkit library uses this library.
  • libz.dylib: A general compression library.
  • MessageUI.framework: Provides the UI for composing emails.
  • OpenGLES.framework: OpenGL for Embedded Systems: Provides classes to create 2D and 3D graphics.
  • Security.framework: Provides decryption support.
  • StoreKit.framework: Provides the classes to deliver In-App Purchase for iOS.
  • SystemConfiguration.framework: Provides functionality to monitor the network state of iOS devices and detect network errors.
  • UIKit.framework: Provides the classes to construct and manage the UI for iOS.

The SDK also depends on this framework, but it should be weak-linked (the optional setting in Xcode):

  • AdSupport.framework: Provides apps with access to an identifier that can be used only for serving advertisements (iOS 6+).
Note
Weak-linked frameworks let you support older versions of the iOS in the same app that uses the features of the latest version. Symbols in weak-linked frameworks do not have to be present at runtime for a process to continue running.

Step 3: Integrate the Big Fish iOS SDK framework

  1. Ensure that you have a clean Xcode project that contains all of the required frameworks and libraries.
  2. Drag the bfg_iOS_sdk.framework folder to the Frameworks folder for your project. You can also use the Add framework UI in Xcode.
  3. Drag the resource bundle for your game to your game resource folder. The resource bundles are located in the framework directory of the SDK.
    • For iPhone only games, use the bfgResourcesiPhone.bundle.
    • For iPad only games, use the bfgResourcesiPad.bundle.
    • For universal games, use the bfgResourcesUniversal.bundle.
    • If your project has targets for both iPhone and iPad, add the iPhone bundle to the iPhone target and the iPad bundle to the iPad target.
  4. Add or change your includes to support the new framework. When including the SDK, your imports must follow this framework convention: #import <bfg_iOS_sdk/bfgManager.h>
  5. Copy the bfg_gamespecific_resources directory to your project’s resources folder. You will customize these assets for your specific game.

Step 4: Customize game-specific resources

The bfglib_gamespecific_resources directory provides an example of resources specific to each game. Developers must include the resources for all supported devices by replacing the example resources with the resources for their game.

Customize these assets for your game as described in Customizing Game-Specific Resources.

Step 5: Initialize bfgManager

Initialize the bfgManager with the viewController that controls rotation for your application. You can change this later with [bfgManager setParentViewController: (UIViewController *) parent]. It is important that you do this in applicationdidFinishLaunchingWithOptions to ensure the initial session data is captured correctly.

applicationdidFinishLaunchingWithOptions:
(NSDictionary *)launchOptions
{
...
[bfgManager initializeWithViewController:
myRootViewController];
...
}

Step 6: Start the implemented services

Depending on what features of the SDK your game supports, you will need to start different services as soon as your application is ready to receive notifications.

If you support In-App Purchase in your application, you should start listening for purchase notifications and then start the bfgPurchase service.

If you support login, you should start listening for the login notification and then start the bfgLoginController service.

Step 7: Register for the resulting notifications

Depending on what functionality you wish to track, you should register for the associated notifications and respond accordingly. Notification are how the SDK communicates to start your application, track UI elements opening and closing, IAP results and other behaviors.

Step 8: Implement Big Fish components in your game

  • Ads: Enables Big Fish to send advertisements to distributed games.
  • Branding: Provides branding functionality for, based on game-specific resources in another library.
  • Game Discovery Network (GDN): The promotional dashboard that loads before a game begins (Freemium only).
  • Game Event Reporting: Connects the game to a usage reporting site for analytics.
  • Freemium In-App Purchases: Class libraries that support In-App Purchases for Freemium games.
  • Free-to-Play In-App Purchases: Class libraries that support In-App Purchases for Free-to-Play games.
  • iSplash newsletter: Promotes by e-mail (Freemium only).
  • Ratings: Enables users to rate games.
  • Tell A Friend: Enables users to send a promotional e-mail message to friends.
  • Login: Enables users to login with their Big Fish Credentials.

For instructions on development tasks and more information about the Big fish components, see Classes and Components.

Sample code

The Big Fish iOS SDK includes a unified sample code project that contains the major Big Fish components.

The main sample can be built and executed as written. You can view the sample as an example or develop a more customized implementation by using the sample as a starting point.

For more detailed debugging, a debug version of the framework is provided.

Step 9: Debugging

Use the debug version of the framework to see more details logged to the console. For more information, see Debugging with the SDK.
