Usability requirements for developing Big Fish Freemium games.

Usability Planning

Game builds must meet several usability requirements before they are accepted for Beta and Release. These should be planned for during the Specification milestone.

Note
Before submitting your game, review Apple's requirements described in App-Related Resources in Apple's iOS App Programming Guide.

Touch Interface

Use these guidelines when developing your game(s) for touch devices:

  • Cursors, mouse-over events, and PC language: Touch interfaces do not require mice or cursors. Remove all references to cursors and mouse-over events, and change all "click" references to "tap". Text should not refer to any interface element that mobile devices do not use.
  • Offset movable objects from the point of touch: To ensure that they remain visible when moved, items should be displaced directly above the player's finger in case the item needs to be used along the edges of the game screen. An option should be included in the Settings sub-menu to reverse for left-handed play.
  • Hit boxes for objects, buttons and interactable areas: The smaller the screen, the more difficult it becomes to accurately touch specific areas. Buttons should be large and easily accessible/tappable. Hit boxes/touch zones should be large enough (often times larger than the object/button/area itself) so that they can be accurately touched.
  • Use photograph manipulation controls for pinch, zoom pinch-zoom, and scrolling: To enable users to enlarge and navigate the hidden object scenes, games should use the same controls that they use to manipulate photographs.
  • Quick response to user input: A game should always respond quickly (nearly instantaneous) to user input, including taps, multi-taps, dragging, pinching, and pinch-zooming on all supported device models.
  • Use photograph manipulation controls for pinch, zoom, and scroll: To enable users to enlarge and navigate the hidden object scenes, games should use the same controls that they use to manipulate photographs.
  • Quick response to user input: A game should always respond quickly to user input, including taps, multi-taps, dragging, pinching, and zooming on all supported device models.

Start Up, Set Up, First Play Session

Games must meet the following requirements for start up, new game set up, and first play sessions.

Profile creation

There is a built-in, on-screen keyboard on iOS devices. The keyboard should automatically open when the player is prompted to create a new profile at the beginning of the game. The profile creation dialog should also appear high enough on the screen so that all buttons are accessible.

Tutorials

Tutorial text should be clear and easy to read. A mechanic should be used to highlight the game element being discussed.

If any changes or additions were made to the UI for mobile (e.g., navigational help button added or the strategy guide button moved for visibility), then new tutorials should be added to teach the player about these new features.

Tutorials should be on by default.

  • In the beginning of the game a prompt should ask the user if they want the tutorial.
  • If "Yes" is selected, an option should be included to turn the tutorial off at any time.
  • OPTIONAL: The in-game options may include a toggle to turn the tutorial on/off at any time.

Game Modes

The dialog at the beginning of the game, outlining game modes available to the player, should be large with clearly readable text.

Some games offer the option to change the difficulty/game modes in-game from the Options sub-menu. To prevent the player from accidentally changing game modes, a pop-up should appear after the player selects a new game mode from Options. This pop-up asks the player to confirm the difficulty/game mode change.

UI and HUD

Games must meet the following requirements for user interface and head-up display elements.

Core buttons

The core button placement should be consistent across titles with Inventory in the bottom middle, Menu to the bottom left, and Hint to the bottom right. The Strategy Guide button should always be clearly visible on the HUD in-game as well.

Positive/negative button placement

Mobile devices are handheld and users treat them almost like they are reading a book. English-speaking users read from left to right and down, which makes the bottom right the last place they will look and where they will spend the most focus. All dialogs with confirmation options should include the positive selection (OK, Yes, checkmarks, etc.) on the right so that users will be more likely to select them.

Close button placement

All overlays, pop-ups, dialogs, and vignettes should have an "X" button in the top right corner. This should be the standard and other close buttons (cancel, close, back, etc.) should not be used. The user should be able to close out of dialogs by touching the "X" button OR touching outside of the dialog.

Full screen zoom zones or scenes where a Back button or back navigation would have been used should also include an "X" to close for easy exit.

Inventory

  • The Inventory menu should not cover up objects or other interactable zones in the scene.
  • The inventory should default to open and locked. If the option is available to unlock the inventory, then this should be called out in the game's tutorial.
  • Inventory items should be highlighted for best visibility.
  • OPTIONAL: If a standard highlight is used, use a different colored highlight for inventory items that can be used in the current area.
  • Navigational areas at the bottom of the screen should be avoided, but if they are used, then they should always be accessible even if the inventory is open.
  • On iOS 7 the multitasker is accessed by swiping up at the bottom of the screen. Make sure that your inventory tab does not interfere with this function.
  • The item name should automatically be shown when the item is selected and be easy to read.
  • The user should be able to select an item and display the item name with a single tap. No double taps or "tap-and-hold" mechanics should be used.
  • Players should be able to use inventory items in game scenes and combine items in hidden object scenes using a "tap-and-drag" mechanic.

User Experience

Games must meet the following requirements to ensure the optimal user experience on touch devices.

Visuals

Critical game elements should not blend in. Icons, items, text, etc., should all pop against the background so they are very easy to see.

Mobile games should have a higher contrast and scenes should be brighter than the PC art.

Text

Text should always be tested on the smallest possible device (e.g., iPhone 4 for iOS) to ensure that text is readable on all device types.

Text should never blend in with text boxes or backgrounds. A contrasting color should always be used, bold wherever possible.

  • It is recommended that text is as close to 24 point as possible for maximum legibility.
  • Embellished fonts are not recommended as they are less readable at smaller sizes.
  • All critical game text (e.g., tutorials, tasks, hints, etc.) should use a standard font. Sans-serif fonts are preferred.
  • Games should use "universal" language wherever possible. This will reduce the amount of changed strings when going from PC to mobile. PC terms (click, cursor, mouse, etc.) should be replaced with universal terms (interact, use, select, activate, etc.).

Item names

  • Items should be labeled correctly. For example, this item was named "cucumber" but the image is of a "pickle."
  • Item names should only apply to one item in a scene.

Puzzles and mini-games

  • Puzzles and mini-games must be skippable.
  • Puzzles and mini-games should include basic operation instructions.
  • On iPhone, if the player taps on operation instructions, a full-screen dialog should appear with the instructions in a larger, more readable font.

Hint and skip recharge

Charge rates should be changed to be more "mobile friendly."

  • Hints should charge within 30 seconds on the easiest game mode. 60 seconds is the maximum time a player should have to wait before getting a new hint on the hardest game mode.
  • Skips should charge within 60 seconds on the easiest game mode. 120 seconds is the maximum time a player should have to wait before being able to skip on the hardest game mode.

Ratings and Reviews

To increase positive ratings in the App Store, it is important that the placement of review prompts is carefully considered. Ask your Big Fish Producer to suggest the best placements according to the following best practices:

  • First Prompt: 15-20 minutes into the game at a "wow moment." (Be sure that the tutorial has been completed.)
  • Second Prompt: Shortly after the Buy Wall. If there is another "wow moment" this would be great, otherwise after about two (2) mini-games and/or hidden object scenes.
  • Third Prompt: After the player has completed the game and before they begin the bonus content.

Saving

Games must meet the following requirements to save, store, and retrieve progress:

  • Save all user-generated game data to an iCloud document or key-value storage: Apple restricts storage to the Documents folder, using key-value storage, or to the Library/Cache. For additional help, see Apple's documentation.
  • Save continuously: Continuously store progress, and return the user to the previous stopping point at the start of each session. Note: Saving progress every few minutes is not often enough in a mobile game.
  • Save hidden objects: Hidden object games must store all found objects as users find them, and the current object list from the most recent scene.
  • Save at every stage: Mini-games that have multiple stages of progress must store progress at each stage.
  • Save on exit: Progress must be saved when users exit by using the home button, Fast App Switching, or when the device goes to sleep, and not just when users exit by using the Main Menu.
  • Save secondary game data: Save game data in two locations to help recover from data corruption in the primary data.
  • Save In-App Purchases: If the user makes an In-App Purchase, save the data immediately and then "finish" the purchase. For more information, see Making Freemium In-App Purchases.
  • Updates to games must be compatible with saved data from previous versions.
Important: Big Fish QA will test compatibility with saved data. Any update build that is not backwards compatible will be rejected.

Orientation and Rotation

Games must meet the following requirements for display orientation and rotation:

  • Rotate between all supported orientations. For Freemium games, only landscape orientation is available.
  • Games with only a portrait view must include both up and down orientations, and be able to rotate between them.
  • Games with landscape orientation must include both right and left orientations, and be able to rotate between them.
  • Rotation should be accompanied by animation.

Loading

When a game loads, it should perform the following tasks:

  • Display screen loading animations: Any screen that takes more than one second to load must display an animation indicating progress, or a loading message that can be accompanied by the default Apple spinner, to indicate that the device/app has not frozen and the screen is still loading. This Loading message should match the theme of the game, but will not be accepted if any developer branding is displayed.
  • Check integrity of primary saved data when loading: If data is corrupt, the game should attempt to load from secondary data.

Menus

Game menus must include buttons and other graphical elements to support the required features. For the required menu elements, see Freemium Main Menu Best Practices.

Icons

Each game has an icon. Icons visually identify the game in the App Store and on the device. The icon for your game will be provided by Big Fish after QA has started. Icons should be included as part of the application bundle for your game. For more information about the contents of the application bundle, see Build-Time Configuration Details in Apple's iOS Application Programming Guide.

When you integrate icons, you must meet the following requirements.

Prevent icon shine

In Xcode (version 4.4.1 and later), go to the build target and on the Summary tab, uncheck "Prerendered" next to "App Icons". This value can also be set manually:

  1. In Xcode, in the Resources folder for your game, open the information property list file info.plist.
  2. Select any item in the list and click the plus (+) sign to create a new list item.
  3. On the key field pick-list, select scriptable type.
  4. Select the Value field checkbox.
  5. In the Key field, click the word scriptable and change its value to UIPrerenderedIcon.
  6. Exit the property editor to save the changes.

Prevent concatenated icon names

Big Fish will provide a Bundle Display Name to be shown under your game's icon on the device springboard and in searches.

Any game title with too many characters will be concatenated. The limit is based on character width and varies by device. Test the title and report back to your Big Fish Producer if a new, shorter title is needed.

For example, for the "Megaplex Madness" game icon, the Bundle Display Name would be shortened to "Megaplex" so it would fit.

Audio

All games must support the following audio options:

  • Volume sliders and knobs in the Options sub-menu default to 70%, not 100%.
  • If the device is set to mute, the game audio must be turned off.
  • Audio played through multi-tasking apps such as Music, Pandora, and Spotify must take precedence. When outside audio is playing, do not play game music.
