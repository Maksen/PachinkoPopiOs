Usability requirements for developing Big Fish Free-to-Play games.

Usability Requirements

Game builds must meet several usability requirements before they are accepted for Beta and Release. These should be planned for during the Specification milestone.

Note
Before submitting your game, review Apple's requirements described in App-Related Resources in Apple's iOS App Programming Guide.

Touch interface

Apps on devices with touch interfaces present common pitfalls. Use these guidelines when developing your game(s) for touch devices:

  • Cursors, mouse-over events, and PC language: Touch interfaces do not require mice or cursors. Remove all references to cursors and mouse-over events, and change all “click“ references to “tap“. Text should not refer to any interface element that mobile devices do not use.
  • UI/HUD: The UI/HUD (head-up display) should be tailored for touch devices so that it is easy for players to use. Buttons should be easily accessible and tappable, all content should be clearly visible, and text labels should be easy to read.
  • Offset movable objects from the point of touch so that they remain visible when moved. An option should be included in the Settings sub-menu to reverse for left-handed play.
  • Use photograph manipulation controls for pinch, zoom, and scroll: To enable users to enlarge and navigate the hidden object scenes, games should use the same controls that they use to manipulate photographs.
  • Quick response to user input: A game should always respond quickly to user input, including taps, multi-taps, dragging, pinching, and zooming on all supported device models.

Orientation and rotation

Games must meet the following requirements for display orientation and rotation:

  • Rotate between all supported orientations.
  • Games with only a portrait view must include both up and down orientations, and be able to rotate between them.
  • Games with landscape orientation must include both right and left orientations, and be able to rotate between them.
  • Rotation should be accompanied by animation.

Loading

When a game loads, it should perform the following tasks:

  • Display screen loading animations: Any screen that takes more than one second to load must display an animation indicating progress, or a loading message that can be accompanied by the default Apple spinner, to indicate that the device/app has not frozen and the screen is still loading. This Loading message should match the theme of the game, but will not be accepted if any developer branding is displayed.
  • Check integrity of primary saved data when loading: If data is corrupt, the game should attempt to load from secondary data.

Saving

Games must meet the following requirements to save, store, and retrieve progress:

  • Save all user-generated game data to an iCloud document or key-value storage: Apple restricts storage to the Documents folder, using key-value storage, or to the Library/Cache. For additional help, see Apple's documentation.
  • Save continuously: Continuously store progress, and return the user to the previous stopping point at the start of each session. Notes: Saving progress every few minutes is not often enough in a mobile game.
  • Save hidden objects: Hidden object games must store all found objects as users find them, and the current object list from the most recent scene.
  • Save at every stage: Mini-games that have multiple stages of progress must store progress at each stage.
  • Save on exit: Progress must be saved when users exit by using the home button, Fast App Switching, or when the device goes to sleep, and not just when users exit by using the Main Menu.
  • Save secondary game data: Save game data in two locations to help recover from data corruption in the primary data.
  • Save In-App Purchases: If the user makes an In-App Purchase, save the data immediately and then "finish" the purchase. For more information, see Making Free-to-Play In-App Purchases.
  • Updates to games must be compatible with saved data from previous versions.
Important: Big Fish QA will test compatibility with saved data. Any update build that is not backwards compatible will be rejected.

Menus

Game menus must include buttons and other graphical elements to support the required features. For the required menu elements, see Free-to-Play Main Menu Best Practices.

Icons

Each game has an icon. Icons visually identify the game in the App Store and on the device. The icon for your game will be provided by Big Fish after QA has started. Icons should be included as part of the application bundle for your game. For more information about the contents of the application bundle, see Build-Time Configuration Details in Apple's iOS Application Programming Guide.

When you integrate icons, you must meet the following requirements.

Prevent icon shine

In Xcode (version 4.4.1 and later), go to the build target and on the Summary tab, uncheck "Prerendered" next to "App Icons". This value can also be set manually:

  1. In Xcode, in the Resources folder for your game, open the information property list file info.plist.
  2. Select any item in the list and click the plus (+) sign to create a new list item.
  3. On the key field pick-list, select scriptable type.
  4. Select the Value field checkbox.
  5. In the Key field, click the word scriptable and change its value to UIPrerenderedIcon.
  6. Exit the property editor to save the changes.

Prevent concatenated icon names

Big Fish will provide a Bundle Display Name to be shown under your game's icon on the device springboard and in searches.

Any game title with too many characters will be concatenated. The limit is based on character width and varies by device. Test the title and report back to your Big Fish Producer if a new, shorter title is needed.

For example, for the "Megaplex Madness" game icon, the Bundle Display Name would be shortened to "Megaplex" so it would fit.

  • Add all icon files to the top level of your game’s application bundle, in the Resources folder.

For more information on Apple requirements for icons, also see App-Related Resources in Apple's iOS App Programming Guide.

Audio

All games must support the following audio options:

  • Volume sliders and knobs in the Options sub-menu default to 70%, not 100%.
  • If the device is set to mute, the game audio must be turned off.
  • Audio played through multi-tasking apps such as Music, Pandora, and Spotify must take precedence. When outside audio is playing, do not play game music.
