The Big Fish iOS SDK and iOS Game Flow on Launch

The following illustrations highlight how the SDK affects a game's UI on launch, and options for loading and landing pages. Big Fish can provide requirements for specific Free-to-Play games. For more information, contact your Big Fish Producer.

Springboard

Game Icon:

  • Big Fish will provide a Bundle Display Name to be shown under your game's icon on the device springboard and in searches.
  • Icons in all sizes will be provided by Big Fish.
  • Initializes bfgManager.
Note
Downloaded server settings may change your local settings.

For more information, see Free-to-Play Usability Requirements and bfgManager.

Calls:

  • [bfgManager initializeWithViewController:myRootViewController] (first time)

The following helper methods are available for more information:

  • [bfgManager isFirstTime]
  • [bfgManager isInitialized]
  • [bfgManager isInitialLaunch]
  • [bfgManager sessionCount]

For Free-to-Play games, the UserID should be set after the user has logged in.

  • [bfgManager setUserID]

Default blue watermark

Watermarks are default images that are provided by your producer.

  • iPad: This default background is a customized blue watermark specific to your game.
  • iPhone: The default background is plain blue. It is dynamically generated, and no image is provided.

Server connections on launch

Note
This architecture also applies to all types of games. A Free-to-Play game would have "dashboard_active" set to 0, and so the Game Discovery Network (GDN), a Freemium game feature, would not display.

The first time a game launches, it reads the settings in the bfg_default_settings.json file to determine whether or not to show the GDN.

On subsequent launches, if the game is able to connect to the server, it reads the settings from the server.

Any time the user connects to the server, default settings may be altered. For example, after the game successfully connects to the server once, the bfg_default_settings.json file may no longer be used.

Calls:

  • [bfgPurchase startService] – Initiates the bfgPurchase process. Start the service when a game is ready to handle a purchase, as soon as possible after launch.
  • [bfgPurchase acquireProductInformation] – For each product offered for purchase in the game. Start right after [bfgPurchase startService]. It runs in the background, and updates current product information.
Note
After a user pauses the game, the dashboard enters a Pause/Resume mode. No additional code is needed to create or initialize this view, and display of this view will be handled automatically by the bfgManager. For more information, see Game Discovery Network.

Branding

Optional.

The bfgbranding_resources.zip file contains the branding.json file that the bfgBrandingViewController class uses for displaying branding images. You can include your own branding images by adding your file names to branding.json.

Calls:

  • [bfgManager startBranding]

Example loading and landing pages

The Big Fish Game logo displays for all games during loading. Use the most recent “splash screen“ image from Big Fish. This image contains the Big Fish logo and no other text, so that the branding does not need to be localized.

It is located in the bfgbranding_resources.zip file.

Once loaded, the landing page displays and the user can immediately begin play. All available games are shown, with options, profile information, and In-App Purchases incorporated.

After the game finishes loading, a landing screen may display that precedes play. In this example, the Found landing page has a Tap to Continue link for game entry.

The landing page for Zombie Zombie Zombie has a Main Menu page that takes the user into a map screen, where play begins.
