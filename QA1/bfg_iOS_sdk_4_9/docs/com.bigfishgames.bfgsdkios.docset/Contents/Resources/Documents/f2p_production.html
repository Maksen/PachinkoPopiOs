Production requirements for developing Big Fish Free-to-Play games.

Big Fish builds, tests, and distributes games created by third-party developers. To facilitate game production, Big Fish requires developers to follow these production standards.

Note
For further information on Apple submission requirements, see App-Related Resources in Apple's iOS App Programming Guide.

Build Requirements

Submitting a game build

Complete the following steps to submit a game build to Big Fish:

  1. Build games with Xcode 4.5 or later. Notify Big Fish of the version of Xcode used.
  2. Configure the project according to the Xcode requirements described in the following sections of this document.
  3. Test the build in a clean environment prior to submission.
  4. Ensure that the game builds and runs on all targeted devices.
  5. For all builds after the Beta, update the JIRA database and note all bug fixes in the comments.
    • Resolve and reassign bugs back to the original "Reporter" for regression.
  6. Submit the build to the trunk of your External Subversion (SVN) repository on https://extsvn.bigfishgames.com.
  7. Provide full Build notes by e-mail to the mobile-builds@bigfishgames.com alias.
Important: Big Fish will reject builds that do not fulfill all of the above requirements. Rejected builds are not eligible for milestone payments.

Server selection

As part of the Big Fish build process, Big Fish QA requires the ability to select which server a game is using. Developers must build hooks into their game to check for the following preprocessor macros that point to the environments Big Fish has set up for them:

  • PROD_SERVER
  • STAGE_SERVER
  • QA_SERVER

Xcode requirements

Use the following standards for your Xcode project:

  • Project name: Use the full name of the game. If the game name is too long, contact your Big Fish Producer to agree on a shorter name.

  • Targets: Use the targets option to contain multiple game SKUs in a single project. iPhone and iPad should be different targets within the same project. For more information about creating targets in Xcode, see Scheme Editing Help - Building Multiple Targets.

  • Target names: To promote consistency, use the following naming conventions for your targets.
    • Free-to-Play iPad games: Name of the game
    • Free-to-Play iPhone/iPod games: Name of the game

  • Provisioning: Before you commit a build to trunk in SVN, set the provision to “default“ for all targets (instructions below), rather than using the provisioning profile for your company. This enables Big Fish to build projects by using our own provisioning profile.

  • Set the default wildcard option: The iPhone Developer profile is the wildcard.

    Selecting this profile automatically populates the Code Signing Identity field with the code signing identity of the local machine. This enables both developers and Big Fish to populate the field with their local code signing identity without having to change this field manually every time.

    1. Open your project in Xcode.
    2. Click Build Settings.
    3. In the Code Signing section, click Code Signing Identity.
    4. Under the Automatic Profile Selector (Recommended) section, select iPhone Developer.

  • Define a custom URL scheme: In Xcode, add the following key to info.plist files for all targets:
<key>CFBundleURLTypes</key>
<array>
<dict>
<key>CFBundleURLName</key>
<string>Bundle_ID</string>
<key>CFBundleURLSchemes</key>
<array>
<string>Bundle_ID</string>
</array>
</dict>
</array>
Bundle_ID is the bundle identifier for your game, for example com.bigfishgames.testapp. The custom URL scheme enables Big Fish to launch apps from within other apps, Web pages, or e-mail messages.

  • ARM 7 optimized architecture: Set the architectures setting in Xcode to ARM-7 Optimized and add the following to your iPhone infop.plist file:
<key>UIRequiredDeviceCapabilities</key>
<array>
<string>armv7</string>
</array>

Bundle IDs and In-App Purchase IDs

Note
Your Big Fish Producer will provide the necessary Bundle IDs and In-App Purchase IDs for all of your game SKUs.

Standard format:

com.bigfishgames.[gamename][store][device][type]

Where [store] is replaced with the publicly traded stock codes for the platform (for example, iOS games use “aapl”), [device] is tablet or phone, and the [type] is free or f2p.

Each Bundle identifier includes a unique PRODUCT_NAME:identifier, which is also known as the "Bundle ID Suffix."

com.bigfishgames.PRODUCT_NAME:identifier

Big Fish uses the PRODUCT_NAME:identifier for tracking and marketing purposes.

Application bundles

Application bundles contain a package of project resource files for your game in a hierarchical directory with the .app extension. The Xcode project for your game will create an application bundle for each target. It is required that the PRODUCT_NAME:identifier (or Bundle ID Suffix) is used for the application bundle name for each target to help differentiate between SKUs.

For example, if the Bundle ID Suffix strings for Game X are:

  • gamexaaplphonefree
  • gamexaapltabletfree

Then the .app files for Game X would be:

  • gamexaaplphonefree.app
  • gamexaapltabletfree.app
Note
It is also required that the Bundle ID Suffix is used as the Product name and Bundle name in the .plist files for each SKU.

JIRA

You must update Big Fish's web-based bug tracking dataset JIRA before you submit each new build. If you did not receive an e-mail message containing account credentials and the location of your JIRA project, contact your Big Fish Producer.

For each resolved bug:

  1. Add a comment including the build version that contains the fix.
  2. Assign the bug to the "Reporter" who originally reported the bug so that person can track the fix for regression.
  3. Resolve the bug with the appropriate disposition, usually "Fixed."

External Subversion (SVN) repository

Big Fish provides Subversion (SVN) on an external server for use by third-party developers for each iOS/Android game. Each repository is externally linked to the latest version of the Big Fish iOS SDK, including documentation, and contains a trunk directory and a folder for assets required for marketing materials to be committed by the developer.

Follow these requirements while checking in to SVN:

  • Use the SVN repository provided by your Big Fish Producer on https://extsvn.bigfishgames.com. If you did not receive an e-mail message containing account credentials and the location of your repository, contact your Big Fish Producer.
  • Commit the full source code and the project file in the trunk directory for your project.
    • Example: if your repository is https://extsvn.bigfishgames.com/svn/game_project/, submit your builds to https://extsvn.bigfishgames.com/svn/game_project/trunk.
    • If possible, place the project file (.xcodeproj) at the top level, and not within a sub-folder.
    • Properly submitted code in the trunk folder looks similar to this:
/trunk/asset or file folder1
/trunk/asset or file folder2
/trunk/asset or file folder3
/trunk/file1
/trunk/file2
/trunk/file3
/trunk/XCode project file
  • After the first commit to SVN, for all new builds only the changes since the last commit are needed. Do not commit the full source every time. Developers should regularly commit their most recent code changes and revisions.

Big Fish uses an internal build system to pull code from the repository trunk, and creates each build internally by using its own provisioning profiles. For more information about tools and processes for building games, see Required Tools and Standards.

Build notes

After each new build has been committed to SVN, build notes are required describing all changes from the previous version and should be sent via e-mail in the following format: 

            To: mobile-builds@bigfishgames.com

            CC: Producer

            Subject:
            [NOTE: If the build is Pre-Beta, this needs to be mentioned at the beginning of the subject line.]
            [Platform: iOS/F2P/Amazon/Google][Full Game Title] [Trunk Revision]

            Body:
            Hello,          

            A new build of [Full Game Title] [Trunk Revision] on [iOS/F2P/Amazon/Google] is available on SVN, please test [SKUs you want tested].

            [SVN URL]
            [Xcode Version]

            [Producer]

            Build notes:
            [All changes made since last build version. Text only for formatting reasons, no images or screenshots.]

            [Any other special information.]

This e-mail message alerts the Build Management Team, Producer, and QA Team leads at Big Fish to the submission.

  • The SVN repository location is required with all build notes.
  • The name of your Big Fish Mobile Producer is required with all build notes.

Versioning

All game builds must include a version number. Use the following standards for your version numbers:

  • Format the version number as three numbers separated by a period ( X.X.X ).
    • The first number indicates a major release.
    • The second number indicates minor updates to an existing release.
    • The final number indicates a revision, typically an interim build that is not released.
  • For Pre-Beta builds, increment the final portion of the version number starting from 0.0.1.
  • Number the first Beta build as 0.1.0. For each build during Beta, increment the second integer. All Beta builds use a 0. x.0 format.
  • When the game reaches the Release candidate build, update the version number to 1.0.0 for submission to Apple.
  • For updates, the third integer will be incremented up to 9, and then the second integer, and so on. It should be difficult to ever reach 2.0.0 for any game title.
  • The version number should be the same everywhere in the game and in the info.plist file.

Localization

Big Fish cannot commit to localizing every iOS game into all supported languages. To achieve the benefits of localization with minimal costs, Big Fish will localize only games that have already been localized or approved for localization for the PC or Mac OS, with the exception of Russian and very occasionally other languages not yet supported for PC. All supported languages should be confirmed with your Big Fish Producer during the specification process.

Localization files are submitted each Thursday for processing the following week. Try to submit earlier in the week to ensure that your location files will be included in the next week's batch.

The following recommendations and requirements help simplify localization and control costs:

  1. Submit all text strings in .txt format.
  2. Begin all lines with a semicolon followed by a String ID for reference. Most developers will just number their String IDs sequentially, for example, GameName_1, GameName_2, and so on.
  3. After the String ID, include a tabbed space. Do not use a regular space.

  4. After the tabbed space include the game text for translation contained inside quotations. Here is what the final strings should look like:

    ;GameName_1 "Pause"
    ;GameName_2 "Restart"

  5. Start with the localization file that Big Fish Games used for the Mac/PC game and include only the differences that have not yet been translated. Common differences include:

    • Text that refers to "click" or "clicking" must be replaced with text that refers to "tapping," and any references to a mouse or cursor, or any other interface element not used by mobile devices, must be removed.
    • Text related to menus and other UI elements in iOS that differ from those on a PC or Mac.
    • Text changes necessitated by the integration of the Game Discovery Network and other Big Fish Games features.
    • Text changes necessitated by the integration of common iOS features and menus such as Game Center, Achievements, or In-App Purchases.

  6. Submit files in English only. Do not submit files that contain multiple languages, or scripts that add language folders.
  7. Allow 1 to 2 weeks from when you submit the text strings in English to your Big Fish Producer for your localization files to be returned. You will receive a set of translated files, one for each language. Merge the changes into the original PC or Mac OS files to make a completed iOS localization package.
  8. After your Producer returns your loction files, create in the Xcode project for your game a language-specific project subdirectory for every language that your game supports. Name these subdirectories in the format language.lproj where "language" is replaced with the standard two-letter code for each language (EN, ES, FR, DE, JP, IT, NL, RU).
  9. If you discover that you have forgotten to submit a text string for localization, or if you make changes that require additional localized strings, you can submit additional requests to your Big Fish Producer. If a game is already in QA, then all localization requests will be handled by the assigned localization QA Test Lead.
  10. Your game should automatically detect the language settings for the device and display the text that matches those settings.
  11. All text must be localized by the time you submit your Beta build.
Important: Any Beta build without localizations, except for games that are English-only, will be rejected.

Art Requirements

To help you maximize your game’s profile and revenue potential by optimizing merchandizing on the Big Fish website and in Appstores, Big Fish requires the following art assets. All should be considered mandatory.

Images

Any background images, landscapes, and scenes from your game without any UI. The higher the resolution, the better! The minimum resolution should be 1024x768, but the maximum available resolution is strongly preferred.

Logos

All game logos, in all forms, on transparent backgrounds. If you use your logo in multiple orientations, include all of them. Please provide these in all languages your game will be released in. Please provide your logos in layered PSD (strongly preferred) or PNG formats.

Game characters

All the characters in your game on transparent backgrounds. Include all variations and each in multiple poses – not just static. Include full lengths, 1/3, and close-ups, in the maximum available resolution. The higher the resolution, the better!

Specific game elements

These are discrete elements of game play on transparent backgrounds. Examples include objects from hidden object scenes, pieces from a puzzle, and blocks/gems from match 3 games.

Loading screens

Any and all loading screens used throughout your game. The higher the resolution the better! The minimum resolution should be 1024x768, but the maximum available resolution is strongly preferred.

Sprites and animated graphics

These are images that make up the animations of your game in a layered PSD or in a collection of PNGs.

UI and fonts

All aspects of the UI and any fonts you use within your game (including logo subtitles). These work great for carrying the theme of your game into Big Fish assets.

Screenshots

Big Fish requires 4 or 5 screenshots of different parts of your game that you would like to feature (no duplicate locations or similar views, please). Hidden object adventure games need to have at least 1 hidden object scene, 1 puzzle scene, and 1 adventure scene. These should be as exciting as possible, and should show off different and unique aspects of the game. If there is any text - for example, object lists for hidden object scenes – Big Fish will need additional localized screens for all languages supported.

All screenshots - in all languages - should be provided in the following sizes:

  • iPad (2048x1536)
  • iPhone (960x640 AND 1136x640)

Miscellaneous art files

Supply anything else you think will help Big Fish make the best assets possible!

Game description text and marketing bullet points

  • Detailed description of the game’s story, style, genre, mechanic (1-2 paragraphs)
  • Specific number of hidden object scenes
  • Specific number of mini-games/puzzles
  • Specific number of chapters/adventure scenes
  • Any unique game play elements
Important: Any Beta build without assets will be rejected. Rejected builds are not eligible for milestone payments.

Apple Demo Build

On rare occasions, Apple may request a demo build for select games for in-store promotions at the Apple Store. Demo builds disable several game features, including all of the features in the Big Fish iOS SDK. Provide a demo build if requested by your Big Fish Producer.

Note
Big Fish passes this build to Apple without any formal testing, unless Apple requests otherwise.
  1. Disable all Big Fish features.
  2. Disable all network features, such as Game Center.
  3. Disable any features that require a user to log in, or that store an account, password, or player name.
  4. If the full application requires a login, the demo app should launch as already logged in with demo credentials that you provide.
  5. You must have legal rights to the demo credentials that you use. For example, do not use the names of licensed characters from movies or other intellectual property that you do not own.
  6. Disable logins to all third-party accounts, such as Facebook and Twitter, and to the online leader board.
  7. For disabled features, include a pop-up message that says, "Feature not enabled for this demo."
    • This message must be localized for all languages that the demo supports. Ask your Big Fish Producer for the localized text for your supported languages.
    • Display the message just prior to the point where the feature is disabled.
  8. When the game is launched, to indicate it is a demo version of the full product, add a splash screen that displays the text, “This version of game_name has been created specifically for demonstration purposes.“ Ask your Big Fish Producer for localized text.
  9. Build your demo and verify that it successfully installs on your development device.
  10. Verify that the icon for the demo game looks exactly like the icon for the full game.
  11. Verify that the demo game has the same name as the official game on the App Store.
  12. In the build notes, include a thorough description of the differences between the demo build and the full build.
  13. Ask your Big Fish Producer to provide translations for any other required demo text.
